## Законы проекта (обязательны к исполнению)

### Закон 1. Один файл — одна ответственность, лимит 500 строк

- Файл растёт выше **500 строк** — ОБЯЗАН предложить разделение **до** написания нового кода
- Новая тема — новый файл, не дописываешь в существующий
- Если при добавлении функции файл **превысит 500 строк** — сначала предложи архитектурное решение по разделению, потом пиши код

**Порог реакции:**
| Строк | Действие |
|---|---|
| < 350 | Пишем, всё хорошо |
| 350–500 | Пишем + предупреждение: «файл приближается к лимиту» |
| > 500 | **Стоп.** Предложить разделение прежде чем писать код |

**Формат уведомления (писать именно так):**
> «Файл `handlers/battle.py` уже 944 строки — выше лимита в 500.
> По регламенту предлагаю разделить на `handlers/pvp.py` (PvP между игроками) и `handlers/pve.py` (бои с ботами).
> Делаем так?»

### Закон 2. Структура "Лего" — каждое новшество независимый блок

- Новая игровая система → новый файл в `repositories/`
- Новый тип контента (мечи, броня, магия) → своя папка или файл **с самого начала**, не подселять к другому
- `database.py` только координирует (импорты + `db = Database()`)
- `config.py` содержит всё что настраивается — туда, не в логику

**Способ "Матрёшка"** — если файл забит под завязку, превращаем его в папку:
```
handlers/battle.py  →  handlers/battle/
                          __init__.py
                          pvp.py      ← игрок vs игрок
                          pve.py      ← игрок vs боты/монстры
                          rounds.py   ← механика раундов
```

**Способ "Библиотека умений"** — логика боя остаётся в одном файле, описания навыков уходят в `skills.py`.
Файл боя «худеет», новые умения добавляются в отдельном месте.

### Закон 3. Чистый импорт — нет циклических зависимостей

Порядок зависимостей (A может импортировать B, B не импортирует A):
```
config.py / progression_loader.py   ← основание, ничего не импортируют из проекта
db_core.py        ← импортирует только config
db_schema.py      ← импортирует db_core, config
repositories/*    ← импортируют db_core, config (не друг друга)
database.py       ← импортирует все repositories (точка сборки)
battle_system.py  ← импортирует database, config
handlers/*        ← импортируют database, battle_system, config
api_server.py     ← импортирует database, battle_system, config
```
Если модулю нужно что-то «выше» по цепочке — данные передаются параметром, не импортом.

### Закон 4. Чистая память — минимальный контекст

Перед правками определить **минимальный набор файлов**, необходимых для задачи.
Не загружать лишние модули, если они не связаны с текущим изменением.
Чинишь PvP → читаешь только `handlers/battle.py` и `repositories/battles.py`, не весь проект.

### Закон 5. Понятный отчёт после каждого изменения

После каждого изменения обязательно писать два блока:

**Технически:**
1. Что изменено
2. В каких файлах
3. Как проверить

**По-человечески** (без терминов и кода):
> Что изменилось в игре для владельца и для игрока — простыми словами.

### Закон 6. Безопасный рефакторинг

При любом разделении файла:
1. Сначала проверить все импорты
2. Восстановить все связи между старым кодом и новым
3. Запустить smoke-тест: `python -c "from database import db; from handlers import BotHandlers"`
4. Проект должен быть рабочим **после каждого сообщения**, не «доделаем потом»

### Закон 7. Лаконичность

- Не переписывать весь файл в чат, если изменена одна строка
- Показывать только изменённый фрагмент + краткий отчёт
- Не добавлять docstring, комментарии и type annotations к коду, который не менялся

### Закон 8. Гигиена кода — нет мёртвого кода

- Если функция или файл больше не используются — предложить удалить
- Не копировать старый код «про запас» — заменять на новый
- Никакой «истории изменений» внутри файла — для этого есть Git
- В файлах только то, что работает в игре прямо сейчас

### Закон 9. Новые категории — новый дом

Если добавляется новый тип предметов или контента (мечи, броня, магия, события) — сразу создавать под них отдельный файл или папку.
Не подселять «Мечи» в файл к «Монстрам». Каждый новый вид контента должен иметь свой дом с самого начала.

### Закон 10. Контроль версий

После каждой значимой правки:
1. Обновить номер версии в `version.py` (1.0.1 → 1.0.2)
2. Сделать git-коммит с номером версии в сообщении
3. Владелец всегда видит, что код обновился

| Запросы по игроку | `repositories/users.py` |
| Запросы по ботам | `repositories/bots.py` |

## Правила для repositories/ (Mixin-паттерн)

```python
# repositories/новая_тема.py
class НоваяТемаMixin:
    def метод(self, ...):
        conn = self.get_connection()
        ...
        conn.close()
```

После создания добавить в `database.py`:
```python
from repositories.новая_тема import НоваяТемаMixin

class Database(DBCore, DBSchema, ..., НоваяТемаMixin):
    ...
```

---

## Правила SQL

- Плейсхолдеры всегда `?` (адаптер переводит в `%s` для PostgreSQL)
- Имена таблиц и колонок: `snake_case`
- Каждый метод сам открывает и закрывает соединение
- Атомарные операции (покупка, награда, списание) — в одной транзакции с `conn.commit()`
- Новая таблица = новая запись в `_apply_migrations()` в `db_schema.py`

## Как добавить новую игровую систему (чеклист)

1. Нужна таблица? → добавить миграцию в `db_schema.py`
2. Создать `repositories/<система>.py` с Mixin-классом
3. Добавить Mixin в `Database` в `database.py`
4. Методы API → в `api_server.py`
5. Telegram-кнопки → в `handlers/`
6. Smoke-тест: `python -c "from database import db; from handlers import BotHandlers"`
7. Обновить версию в `version.py`, сделать git-коммит

## Чего нельзя делать (запрещено)

- Писать новый код в `database.py` — только импорты
- Смешивать темы в одном файле
- Дублировать SQL-запросы
- Добавлять бизнес-логику в `db_schema.py`
- Создавать циклический импорт
- Дублировать имена методов между Mixin-классами
- Хардкодить токены, URL, пароли — только через `.env`
- Оставлять мёртвый код (неиспользуемые функции, файлы)
- Копировать старый код «рядом» с новым — только заменять
- Переписывать весь файл в чат при изменении одной строки

## Дополнения владельца (дополнение к регламенту; пункты выше не отменяются)

### Высшая инструкция — усиление

Этот файл (`CLAUDE.md`) — **высшая инструкция для ИИ**. Работать по ней **всегда**, не заставлять владельца напоминать. **Всегда следовать ей.**

### Как проект разбит для навигации (простыми словами)

Цель: **ИИ в проекте ориентируется идеально**, а владелец имеет **простую карту** и не путается.  
Подробная карта папок и файлов — в разделе «Структура проекта (карта)» выше; новый код кладём **только в свои тематические места**, а не «всё подряд в один файл».

### Работа после рефакторинга: строго в новых тематических файлах

После того как логика разнесена по темам (`repositories/`, `handlers/` и т.д.), **работать строго внутри этих новых файлов и папок**.  
Не наращивать «старые» монолиты снова, если для темы уже есть отдельный модуль.

### Дополнительный аварийный порог: 1000 строк

Если файл `handlers/battle.py`, `handlers/commands.py` или любой другой крупный файл снова **растёт выше 1000 строк** — **сразу** предложить владельцу его разделить (как именно — по смыслу, например PvP vs PvE, команды vs колбэки), **до** продолжения «наращивания» в том же файле.

Это **дополняет** базовый закон про **500 строк** и **не отменяет** его: при **500+** уже действует «стоп и предложи разделение».

### Пример «распила» боёв (имена под проект)

Допустимые варианты имён модулей (пример): `pvp_battle.py` и `pve_battle.py` (игроки против игроков и игроки против монстров/ботов). Альтернатива, уже встречающаяся в регламенте выше: `pvp.py` / `pve.py`. Папка вместо одного файла — по **способу «Матрёшка»** в Законе 2.

### Способ «Матрёшка» — примеры имён файлов (для Duel Arena)

Сама логика боя остаётся в одном месте, а описания супер-ударов, магии и т.п. выносятся в отдельный модуль (например `skills.py` или аналог), чтобы файл боя «худел», а новые умения можно было добавлять без раздувания ядра.
